ANÁLISIS DE ARQUITECTURA Y COMPONENTES - TRABAJO DE GRADO
================================================================================

1. ARQUITECTURA GENERAL
--------------------------------------------------------------------------------
* Framework principal utilizado: Spring Boot (versión 4.0.6 / Spring Framework 6+).
* Patrón arquitectónico: Arquitectura en capas (Controlador -> Servicio -> Repositorio -> Base de Datos) con Spring MVC para las vistas dinámicas.
* Tecnologías utilizadas en el backend: Java 24, Spring MVC, Spring Security, Spring Data JPA.
* Tecnologías utilizadas en el frontend: Thymeleaf, HTML5, Vanilla CSS, Vanilla Javascript y Phosphor Icons.
* Base de datos utilizada: MySQL / MariaDB.
* ORM utilizado: Hibernate (por medio de Spring Data JPA).
* Librerías de seguridad utilizadas: Spring Security (Core, Web, Config) y JSON Web Token (io.jsonwebtoken: jjwt-api, jjwt-impl, jjwt-jackson).


2. MÓDULOS FUNCIONALES
--------------------------------------------------------------------------------
Los módulos implementados en el sistema son:
* Autenticación: Registro de cuentas, verificación de tokens JWT y login basado en roles de usuario.
* Dashboard: Visualización de métricas en tiempo real (clientes, deshidratadores, compostaje procesado y alertas recientes).
* Gestión de Usuarios: CRUD completo de usuarios del sistema administrativo.
* Gestión de Clientes: CRUD completo y validaciones de clientes.
* Gestión de Perfiles: Mantenimiento de perfiles y roles del sistema.
* Gestión de Registros: Control del ciclo de deshidratación de compostaje (Inicio, Cierre y telemetría histórica).
* Gestión de Sensores: Visualización de lecturas y estados en tiempo real.
* Reportes: Descarga de consolidados operativos diarios, semanales y de alertas.
* Recuperación de Contraseña: Envío de códigos temporales por SMTP (Gmail).

8. SEGURIDAD
--------------------------------------------------------------------------------
* Marco de trabajo: Spring Security.
* Autenticación:
  - Interfaz web: Formulario basado en sesión tradicional.
  - Endpoints REST: Filtrado mediante Token JWT. El filtro "JwtAuthenticationFilter" intercepta peticiones que lleven la cabecera "Authorization: Bearer <token>", valida firma y expiración a través de "JwtService" y autentica al usuario en el contexto de seguridad.
* Clases involucradas: SecurityConfig, JwtAuthenticationFilter, JwtService, CustomUserDetailsService, DaoAuthenticationProvider.
* Protección de rutas: Rutas estáticas de interfaz y endpoints específicos de registro y telemetría de Arduino están permitidos sin autenticar. Las rutas críticas (usuarios, perfiles, etc.) están estrictamente protegidas según el rol asignado (ej. ADMIN).


9. API REST
--------------------------------------------------------------------------------
* POST /api/auth/login: Autenticación de credenciales; devuelve token JWT.
* POST /api/sensores/lectura: Recepción de variables desde Arduino.
* GET /api/registros/{id}/lecturas: Histórico de temperatura/humedad para dibujar la gráfica de Chart.js.
* GET /api/dashboard/stats: Datos consolidados del panel principal.


10. INTEGRACIÓN CON HARDWARE
--------------------------------------------------------------------------------
* Flujo de entrada: El módulo Arduino Uno realiza peticiones HTTP POST con cargas útiles de tipo JSON.
* Receptor: Endpoint POST "/api/sensores/lectura" en "ArduinoTelemetriaController".
* Procesamiento: El controlador asocia las lecturas a los sensores e identifica si hay un ciclo de deshidratación con estado "ACTIVO" enlazado a ese Arduino. Si es así, actualiza los valores del ciclo en tiempo real y, en caso de superar el umbral térmico del tipo de compostaje, crea y persiste una "Alerta".
* Almacenamiento: Se persiste a través de "SensoresRepository" (lecturas) y "AlertaRepository" (alertas).
* Entidades involucradas: Arduino, Deshidratador, Registro, Sensores, TipoSensor, Alerta, TipoCompostaje.


11. COMPONENTES EXTERNOS
--------------------------------------------------------------------------------
* Arduino Uno: Sí, gestiona y remite lecturas de telemetría.
* Sensor de Temperatura/Humedad (DHT11 o similar): Sí, captura variables físicas en la cámara de secado.
* Servidor de Correo (SMTP / Gmail): Sí, envía tokens de restablecimiento de contraseña mediante "JavaMailSender".
* MQTT / WebSockets / APIs Externas: No se utilizan. El flujo de telemetría a UI se realiza por HTTP POST y polling AJAX periódico.


12. PERSISTENCIA
--------------------------------------------------------------------------------
* Motor de BD: MySQL / MariaDB (puerto 3307 por defecto).
* ORM: Hibernate (vía JPA).
* Flujo: Controller (recibe la petición HTTP/JSON) -> Service (gestiona la transacción y lógica de negocio) -> Repository (interactúa con la BD a través de interfaces JPA) -> MySQL.


13. FLUJO DE NAVEGACIÓN GENERAL
--------------------------------------------------------------------------------
1. El usuario inicia sesión en la aplicación web mediante el frontend (Thymeleaf/JS).
2. El frontend realiza peticiones al backend (Spring Boot) para recuperar las vistas dinámicas o consumir APIs REST.
3. Paralelamente, el hardware Arduino envía telemetría periódicamente al backend a través de peticiones HTTP POST.
4. El backend procesa las solicitudes, interactúa con la base de datos MySQL para leer/escribir información y actualiza el estado de las interfaces de usuario.


14. DASHBOARD
